# Architecture

## Stack
- **Frontend:** Next.js 14 (App Router) on Vercel
- **Database + Auth:** Supabase (Postgres + RLS + Auth)
- **AI:** OpenAI GPT-4o via server-side API route
- **Styling:** Tailwind CSS

## What to Build Now vs Later
**Now:** Brief form → AI generation → script viewer → edit sections → usage log → scripts list
**Next:** Auth, per-user history, versioning, PDF export, script ratings
**Later:** Video pipeline integration, storyboard view, team collaboration, usage quotas

## Key User Action — Step by Step
1. User fills brief form (title, topic, tone, audience, key messages) → POST saves `script_brief` row
2. User clicks **Generate Script** → Next.js API route reads brief, calls OpenAI, parses JSON response
3. API inserts `generated_script` row + three `script_section` rows (hook, body, CTA) with confidence scores
4. API inserts `usage_event` row (type: `script_generated`, tokens used)
5. Client redirects to `/scripts/[id]` — fetches script + sections from Supabase, renders in UI
6. User edits a section inline → PATCH updates `script_section.content` + sets `review_status = 'reviewed'`

## Layer Plan
1. **Data first** — tables + RLS + seed rows → app renders without AI
2. **App logic** — form, API route, script viewer, edit, list all work against live DB
3. **Smart layer** — OpenAI call slotted into step 2's API route; if it fails, error is shown, no data corrupted

## Why the Core Runs Without AI
All data is stored in Postgres. The script viewer, editor, and list work on any row — whether generated by AI or manually inserted as seed data. Disabling the OpenAI call breaks generation only; everything else stays functional.
